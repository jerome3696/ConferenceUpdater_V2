# HARNESS_MASTER — 사람용 레퍼런스

> **이 문서는 사람(프로젝트 오너)이 읽기 위한 자료다.**  
> AI가 실행 시 참조하는 문서는 `docs/dev-guide-v2.md`.  
> 여기에는 하네스 엔지니어링 원칙, MVP 평가, 논의 과정에서 나온 의사결정을 기록한다.

> **출처**: OpenAI "Building software with LLMs" 문서 + 유튜브 해설 영상  
> **작성일**: 2026-04-17  
> **대상 프로젝트**: ConferenceFinder (MVP v1.0.0 완료 시점)

## 1. 하네스 엔지니어링이란

### 1.1 배경: 프롬프트 → 컨텍스트 → 하네스

| 단계 | 핵심 질문 | 비유 |
|---|---|---|
| 프롬프트 엔지니어링 | AI에게 어떻게 **말을 잘** 하나? | 신입에게 업무 지시를 명확하게 |
| 컨텍스트 엔지니어링 | AI에게 **일의 히스토리**를 어떻게 전달하나? | 6개월 인턴이 회사 맥락을 아는 것처럼 |
| 하네스 엔지니어링 | AI가 **규칙을 어겼을 때** 어떻게 자동으로 잡아내나? | 강아지 하네스 — 바운더리 설정 + 제동 장치 |

### 1.2 왜 하네스인가

- AI의 코드 생산 속도는 빠르지만, 인간의 테스트/검수가 병목
- 인간 개입을 최소화하려면 AI가 스스로 실수를 잡아내는 시스템이 필요
- "뭐뭐 하지 마"라고 프롬프트에 쓰는 것만으로는 부족 — AI는 무시할 때가 많음
- 핵심: **규칙 위반 시 기계적으로 제동이 걸리는 장치**를 만드는 것

### 1.3 유튜브 영상이 제시하는 6대 원칙

#### 원칙 1: 가시성 확보 (AI에게 눈 달아주기)

- AI가 자기 작업 결과를 직접 볼 수 있어야 함
- 방법: 로그 시스템, 스크린샷, DOM 스냅샷, 브라우저 DevTools 연결
- AI가 로그를 보고 "이 결과가 내가 원한 게 아니네" → 스스로 피드백 → 재수정
- 기존 개발에서는 "쓸데없는 로그 지워"가 관행이었지만, AI 시대에는 오히려 로그를 붙여줘야 함

#### 원칙 2: 컨텍스트 목차화 (Progressive Disclosure)

- 하나의 큰 MD에 모든 규칙을 때려넣으면 AI가 앞부분만 읽고 뒷부분은 무시
- 해결: MD 파일을 목차(index)로만 쓰고, AI가 필요할 때 해당 문서를 찾아가게 함
- 진입점 → 목차 → 세부 문서로 점진적 공개

#### 원칙 3: CI/CD 강제 (핵심 중의 핵심)

- 프롬프트로 "이거 하지 마"라고만 쓰면 AI가 무시할 수 있음
- 실제 코드로 강제해야 함: ESLint 규칙, pre-commit hook(Husky), CI 파이프라인
- 예시: 커밋 전에 lint/test/build 자동 실행 → 실패하면 커밋 차단
- "말로만 하지 마" → "기계로 강제하라"

#### 원칙 4: 워크트리 격리

- 메인(main) 브랜치에서 직접 작업하지 않음
- Git worktree를 별도로 파서 거기서 작업 → 테스트 통과 → 메인에 merge
- 복사본에서 완벽하게 만든 후 합치는 전략

#### 원칙 5: 플랜 문서 우선 + 강제 Hook

- 코드 작성 전에 반드시 계획 문서를 먼저 작성
- 계획 문서 → active 폴더에 저장 → 완료 후 completed 폴더로 이동
- **핵심 장치**: 커밋 시 계획 문서가 없으면 hook이 차단
- 구현: 브랜치명에 PLAN ID 포함 → pre-commit hook이 `docs/plans/active/`에서 해당 ID 파일 존재 확인 → 없으면 차단
- 예외: fix/*, docs/*, chore/* 브랜치는 플랜 불필요
- 완료 시: post-merge hook이 자동으로 active → completed 이동

#### 원칙 6: 자동 커밋/머지 + 로그 축적

- AI가 작업 완료 후 직접 커밋, 커밋 메시지도 규칙에 맞게 자동 생성
- 테스트 로그, 스크린샷, 결과 JSON 등을 logs/ 디렉토리에 축적
- 나중에 로그를 보고 회고/개선 가능

## 2. 원칙 vs ConferenceFinder MVP: 1:1 비교

### 비교 요약

| 원칙 | 적용 상태 | 점수 | 설명 |
|---|---|---|---|
| 1. 가시성 (로그/눈) | 부분 적용 | ★★☆☆☆ | PROMPT_LOG + eval 결과 JSON 존재, 하지만 AI 자동 피드백 루프는 없음 |
| 2. 컨텍스트 목차화 | 잘 적용 | ★★★★☆ | CLAUDE.md → blueprint/dev-guide/STRUCTURE 등 분리, 목차 역할 |
| 3. CI/CD 강제 | 미적용 | ☆☆☆☆☆ | pre-commit hook, lint, 테스트 자동화 전무. 모든 규칙이 프롬프트에만 존재 |
| 4. 워크트리 격리 | 미적용 | ☆☆☆☆☆ | main 브랜치 직접 작업 + 직접 커밋. 브랜치 전략 없음 |
| 5. 플랜 문서 우선 | 부분 적용 | ★★★☆☆ | dev-guide Step 구조가 계획 역할, 하지만 기능별 플랜 문서와 강제 hook 없음 |
| 6. 자동 커밋/로그 축적 | 부분 적용 | ★★☆☆☆ | GitHub 자동 커밋 있음(앱 내), eval 결과 축적 있음. 커밋 메시지 규칙 미강제 |

### 원칙별 상세

#### 원칙 1: 가시성 — ★★☆☆☆

**된 것**: eval-prompt.js로 프롬프트 품질 정량 평가 (17건 골든셋), 결과 JSON 저장, PROMPT_LOG에 패턴 분석.

**안 된 것**: AI가 코드 변경 후 스스로 빌드/테스트하는 루프 없음. UI 스크린샷 비교 없음. eval 실행이 수동.

#### 원칙 2: 컨텍스트 목차화 — ★★★★☆

**된 것**: CLAUDE.md가 진입점 + 목차. blueprint(What), dev-guide(How), STRUCTURE(코드 맵) 역할 분리.

**안 된 것**: 문서가 8개로 늘어남 → 확장 시 관리 어려움. 문서 간 상호 참조의 기계적 검증 없음.

#### 원칙 3: CI/CD 강제 — ☆☆☆☆☆

**된 것**: 없음. 모든 규칙이 CLAUDE.md 텍스트에만 존재.

**안 된 것**: pre-commit hook, ESLint 강제, 단위 테스트, CI lint/test 단계, API 키 패턴 감지 — 전부 없음. "파괴적 명령 금지"가 프롬프트에만 있고 코드로 강제하지 않음.

#### 원칙 4: 워크트리 격리 — ☆☆☆☆☆

**된 것**: 없음. main에서 직접 작업, 앱 내 자동 커밋이 미완성 상태를 push할 수 있음.

#### 원칙 5: 플랜 문서 우선 — ★★★☆☆

**된 것**: blueprint + dev-guide Phase/Step 구조. "바로 코딩하지 말고 확인받아라" 규칙.

**안 된 것**: 기능별 개별 플랜 문서 없음. 플랜 없이 코드 커밋해도 차단 없음.

#### 원칙 6: 자동 커밋/로그 축적 — ★★☆☆☆

**된 것**: 앱 내 GitHub 자동 커밋. eval 결과 축적. deploy.yml 자동 배포.

**안 된 것**: 커밋 메시지 규칙 미강제. 테스트/작업 로그 자동 축적 없음.

## 3. Post-MVP 논의 기록

### 3.1 docs 재편 — 합의된 구조

기존 문서를 아래와 같이 재편한다:

```
docs/
├── blueprint.md          ← 기능 정의 (What): 구현됨 / 미구현 구분
├── structure.md           ← 코드 구조 (How): 파일 맵, 모듈 의존 관계
├── design.md              ← UI/디자인 결정 로그
├── prompteng.md           ← 프롬프트 전략 + 실행 로그 통합 (기존 PROMPT_LOG + PROMPT_STRATEGY 합침)
├── changelog.md           ← 버전별 변경 이력
├── dev-guide-v2.md        ← Post-MVP 실행 가이드 (AI가 보는 문서)
├── qa-backlog.md          ← 사용 중 관찰한 이슈 축적 → 일괄 수정용
├── eval/                  ← 골든셋, 결과 JSON
├── plans/                 ← 플랜 문서
│   ├── active/
│   └── completed/
├── reference/             ← 사람용 레퍼런스 (AI 실행과 무관)
│   └── HARNESS_MASTER.md  ← 이 파일
└── legacy/                ← 역할 끝난 문서 아카이브
    ├── dev-guide.md
    └── rate_limit_strategy.md
```

**핵심 결정 사유**:
- blueprint: "프론트엔드"가 아니라 "기능 정의(What)". 서버가 생겨도 기능은 여기.
- structure: "백엔드"가 아니라 "코드 구조(How)". 프론트/백 모두 여기.
- prompteng: PROMPT_LOG(실행 기록)와 PROMPT_STRATEGY(전략)는 같은 주제라 합침.
- legacy/: rate_limit_strategy(해결됨), dev-guide v1(MVP 완료)은 아카이브.
- reference/: HARNESS_MASTER처럼 사람이 읽는 배경 자료.

### 3.2 Sub-agent 구상 — 논의 결과

#### 구상 1: 프롬프트 튜닝 에이전트

- **사용자 희망**: 수동/자동 등 여러 방식으로 업데이트·검증 프롬프트 개선
- **평가**: 시기상조. 골든셋 17건 수준에서는 사람이 직접 v2 써보는 게 빠름. PROMPT_STRATEGY.md에서도 같은 결론.
- **합의**: v2 프롬프트를 사람이 2~3회 수동 이터레이션 먼저. 결과 10회 이상 쌓이면 반자동 log analyzer 도입. 자동 채택 루프는 절대 금지.
- **실행 시점**: dev-guide-v2 Track B 후반

#### 구상 2: 리팩토링/정리 에이전트

- **사용자 희망**: 주기적으로 코드 구조를 보고 개선 제안 + 리팩토링
- **평가**: 가장 실용적. 다만 "에이전트"보다는 체크리스트 + 스크립트가 적합.
- **합의**: `scripts/refactor-check.sh`(파일 줄 수, 복잡도, 중복 감지) 만들고, CLAUDE.md에 "리팩토링 요청 시 먼저 이 스크립트 실행 후 결과 기반으로 제안할 것" 추가.
- **실행 시점**: dev-guide-v2 Track A 완료 후

#### 구상 3: 프론트엔드 QA 에이전트

- **사용자 희망**: 개선사항을 툭툭 던지면 모아뒀다가 일괄 수정
- **평가**: 아이디어 좋음. 별도 에이전트 불필요, qa-backlog.md + 배치 처리 워크플로우로 충분.
- **합의**: `docs/qa-backlog.md`에 이슈를 한 줄씩 축적. 5~10건 모이면 "qa-backlog 처리해줘" → AI가 읽고 플랜 작성(H2.2) → feature branch에서 일괄 수정 → verify-task 실행.
- **실행 시점**: 즉시 (qa-backlog.md 생성은 docs 재편 때)

### 3.3 우선순위 — 합의된 실행 순서

메타 작업(하네스, 문서, 에이전트)에 시간을 과도하게 쓰지 않는다는 원칙.

1. **지금**: docs 재편 + qa-backlog.md 생성
2. **지금**: 하네스 뼈대 (Husky, 테스트, CI, verify-task)
3. **이후**: 실제 기능 개발 시작 (멀티유저, 학회 검색 등)
4. **개발하면서**: qa-backlog 축적, 프롬프트 v2 수동 이터레이션
5. **데이터 쌓인 후**: 프롬프트 반자동화, 리팩토링 체크 스크립트

### 3.4 dev-guide-v2 설계 — 합의된 구조

dev-guide v1과 달리 일직선 순서가 아닌 **병렬 트랙** 구조:

- **Track A: 체계화** — docs 재편 + 하네스 뼈대 + 워크플로우 격리
- **Track B: 코드 품질** — 리팩토링 + 테스트 + 프롬프트 개선
- **Track C: 기능 확장** — 멀티유저, 학회 검색, 기타 신규 기능

트랙 간 의존성: "Track C 시작 전에 Track A 최소 완료 필수"

상세 Step은 `docs/dev-guide-v2.md`에 체크박스로 관리.
