## 1. 목표 (What)

한두 문장. 이 작업이 끝났을 때 무엇이 달라져 있어야 하는가. 측정 가능한 형태로.

## 2. 배경·동기 (Why)

이 작업이 왜 지금 필요한가. 어떤 문제·기회·요청에서 출발했나. 근거 문서·이슈·관찰 사례 링크.

## 3. 범위 (Scope)

### 포함
- 명확히 이 플랜 안에서 처리할 항목들

### 제외 (Non-goals)
- "이건 이번 작업에서 안 한다"를 미리 못박기. scope creep 방지.

## 4. 설계 결정

핵심 판단과 그 이유. 대안을 검토했다면 왜 선택/탈락시켰는지. 코드 수준 디테일이 아니라 방향성.

## 5. 단계 (Steps)

- [ ] Step 1 — ...
- [ ] Step 2 — ...
- [ ] Step 3 — ...

각 단계는 한 커밋 단위로 가능한 정도로 잘게.

## 6. 검증 (Verification)

이 작업이 "완료"라고 말하기 위해 통과해야 할 조건들.

- [ ] `bash scripts/verify-task.sh` 통과
- [ ] (해당 시) 테스트 추가/수정: ...
- [ ] (UI 변경 시) 브라우저 수동 확인 시나리오: ...
- [ ] (프롬프트 변경 시) `npm run eval` 결과 비교, `prompteng.md` §5 갱신
- [ ] (구조 변경 시) `structure.md` 갱신

## 7. 리스크·롤백

이 변경이 깰 수 있는 것. 문제 발생 시 롤백 방법 (revert 커밋, feature flag, 원본 파일 보존 등).

## 8. 후속 (Follow-ups)

이 플랜에서 분리해 후속으로 미룬 항목들. 별도 플랜으로 만들 가치 있음.

## 9. 작업 로그

날짜순으로 짧게. 막힌 곳, 방향 전환, 중요한 발견.

- YYYY-MM-DD: ...
