# 학회 DB 관리 웹앱 — 프로젝트 청사진 (Blueprint)

> **문서 버전**: v1.0  
> **최종 수정일**: 2026-04-14  
> **목적**: 이 문서는 프로그램의 "무엇(What)"을 정의합니다. 새로운 개발 세션에서 이 문서를 제공하면 Claude가 프로그램의 전체 구조를 이해할 수 있어야 합니다.

## 1. 프로젝트 개요

### 1.1 프로젝트 목적

열유체·건물공조 분야의 국제 학회 및 박람회 정보를 체계적으로 관리하는 웹 애플리케이션.  
핵심 가치는 **AI(Claude API)를 활용한 학회 정보 자동 업데이트**이며, 누구나 온라인으로 학회 DB를 열람할 수 있어야 한다.

### 1.2 사용자 및 환경

| 항목 | 내용 |
|------|------|
| 주 사용자 | 연구원 (관리자 1인 + 열람자 다수) |
| 접근 방식 | 웹 브라우저 (PC 중심, 모바일은 대상 아님) |
| 관리자 역할 | DB 편집, AI 업데이트 실행, 데이터 배포 |
| 열람자 역할 | 온라인에서 학회 DB 조회 (읽기 전용) |

### 1.3 핵심 제약 조건

1. **유료 서비스는 Claude API만 사용** — 별도 서버, DB 서비스, 기타 유료 API 사용 불가
2. **온라인 읽기 접근 필수** — 누구든 URL로 접속하면 학회 리스트를 볼 수 있어야 함
3. **서버 없는 정적 호스팅** — GitHub Pages 또는 Vercel 등 무료 정적 호스팅 사용
4. **API 키 관리** — 사용자가 본인의 Claude API 키를 브라우저에서 입력하는 방식

## 2. 데이터 구조

### 2.1 설계 원칙

원본 엑셀에서는 "금번 학회"와 "지난 학회"가 고정 컬럼이었으나, 웹앱에서는 **학회 마스터 정보**와 **개최 이력**을 분리하여 시간이 지나도 구조 변경 없이 이력이 누적되도록 설계한다.

### 2.2 학회 마스터 테이블 (conferences)

학회 자체의 고정 정보. 개최 시기와 무관하게 변하지 않는 속성.

| 필드명 | 타입 | 설명 | 예시 |
|--------|------|------|------|
| id | string | 고유 식별자 (자동 생성) | "conf_001" |
| starred | number | 중요도 마킹 (0~3, 0=미지정) | 2 |
| category | string | 분류 | "학회" / "박람회" |
| field | string | 분야 | "열전달" / "냉동공조" / "유체역학" 등 |
| abbreviation | string | 약칭 (없으면 빈 값) | "IHTC" |
| full_name | string | 정식 명칭 | "International Heat Transfer Conference" |
| cycle_years | number | 개최 주기 (년 단위, 0=비정기) | 4 |
| duration_days | number | 기간 (일 수) | 5 |
| region | string | 지역권 | "세계" / "유럽" / "미주" / "아시아" |
| official_url | string | 공식 위원회/시리즈 URL | "https://www.astfe.org/conferences/" |
| note | string | 참고 메모 (개인 기록) | "김우경박사님 추천" |

### 2.3 개최 이력 테이블 (editions)

특정 학회의 각 회차별 개최 정보.

| 필드명 | 타입 | 설명 | 예시 |
|--------|------|------|------|
| id | string | 고유 식별자 | "ed_001" |
| conference_id | string | 학회 마스터 ID (외래키) | "conf_001" |
| status | string | 상태 | "upcoming" / "past" |
| start_date | string | 시작일 (YYYY-MM-DD) | "2026-08-10" |
| end_date | string | 종료일 (YYYY-MM-DD) | "2026-08-15" |
| venue | string | 장소 | "Rio de Janeiro, Brazil" |
| link | string | 해당 회차 공식 링크 | "https://ihtc18.org/" |
| source | string | 정보 출처 | "ai_search" / "user_input" |
| updated_at | string | 마지막 업데이트 시각 | "2026-04-14T09:00:00Z" |

### 2.4 상태 전환 로직

```
[upcoming] ---(종료일이 오늘 이전)---> [past]  ← 자동 (프론트엔드 로직, AI 호출 없음)
[past/없음] ---(AI 업데이트)---> [upcoming]    ← AI 웹 검색으로 다음 개최 정보 탐색
```

- 종료일이 지난 upcoming은 **자동으로 past로 전환** (페이지 로드 시 날짜 비교)
- 이 전환에는 API 호출이 필요 없음 (비용 0)
- past edition은 AI 검색으로 찾지 않음 (upcoming을 밀어넣어 채우거나 수동 입력)

### 2.5 데이터 저장 방식

| 저장소 | 용도 | 접근 |
|--------|------|------|
| GitHub 저장소 내 JSON 파일 | 기본 학회 데이터 (배포용) | 누구나 읽기 가능 |
| 브라우저 localStorage | 관리자의 수정사항 (임시) | 해당 브라우저에서만 |
| JSON 다운로드/업로드 | 수정사항 영구 반영 | 관리자가 GitHub에 커밋 |

**흐름**:
1. 앱 로드 → GitHub에서 기본 JSON 불러옴 → 화면에 표시
2. 관리자가 편집/업데이트 수용 → 브라우저 localStorage에 저장
3. 관리자가 "데이터 내보내기" → JSON 파일 다운로드
4. 관리자가 해당 JSON을 GitHub 저장소에 커밋 → 배포 데이터 갱신
5. 다른 열람자가 접속하면 갱신된 데이터를 봄

## 3. 기능 정의

### 3.1 기능 1: 정합성 검증

**목적**: 학회 마스터 정보(Conference Info)가 사실과 부합하는지 AI가 검증.

**사용 빈도**: 낮음 (초기 DB 구축 시, 또는 의심되는 항목 확인 시)

**검증 대상**:
- 개최 주기(cycle_years)가 맞는지
- 기간(duration_days)이 맞는지
- 지역(region) 분류가 적절한지
- 약칭(abbreviation)이 올바른지
- 공식 URL(official_url)이 유효한지, 올바른 사이트인지

**동작 방식**:
1. 사용자가 개별 학회의 "검증" 버튼 클릭
2. Claude API 호출 (웹 검색 도구 사용)
3. 검증 결과를 표시: 각 항목별로 "일치" / "불일치 (수정 제안: ○○)" / "확인 불가"
4. 사용자가 수정 제안을 수용/리젝

**내부 구현**: 업데이트 기능과 같은 API 호출 파이프라인을 공유하되, 프롬프트만 다름.

### 3.2 기능 2: 학회 정보 업데이트 (핵심 기능)

**목적**: 다가오는 학회(upcoming)의 개최 정보를 AI 웹 검색으로 채우거나 갱신.

#### 3.2.1 개별 업데이트

- 특정 학회 옆의 "업데이트" 버튼 클릭
- **조건 없이 무조건 검색** (pass 로직 없음)
- 모든 필드를 다시 찾아서 결과 표시
- 사용자가 수용/리젝 판단

#### 3.2.2 전체 업데이트

- 메인 화면의 "전체 업데이트" 버튼 클릭
- 전체 학회 중 **업데이트 대상**을 자동 판별하여 순차 검색

**업데이트 대상 판별 (pass/검색 로직)**:

```
검색 안 함 (pass) 조건 — 아래 조건을 모두 만족해야 함:
  1. upcoming edition이 존재함
  2. start_date, end_date, venue, link가 모두 채워져 있음
  3. source가 "user_input"이 아님 (즉, AI가 이미 채운 정보)

검색함 조건 — 위 조건 중 하나라도 불만족:
  - upcoming이 없거나 비어있는 필드가 있음
  - 또는 source가 "user_input"인 경우 → 링크를 찾아 채워넣어야 함
```

**전체 업데이트 흐름**:
1. 전체 업데이트 버튼 → 업데이트 현황 페이지(페이지 2)로 이동
2. 대상 학회 목록과 총 개수 표시 (예: "10개 학회 업데이트 예정")
3. 순차적으로 AI 호출, 진행률 표시 (예: "3/10 — IHTC 검색중...")
4. 검색 완료된 항목은 "업데이트 대기" 리스트에 추가
5. 각 항목마다 변경사항 표시 + 수용/리젝 버튼
6. 수용된 항목은 "수용 완료" 리스트로 이동 (실제 데이터 반영)
7. **중단 버튼** 제공 — 중간에 멈출 수 있음

**source 처리**:
- AI가 찾아 채운 정보: source = "ai_search"
- 사용자가 수동 입력한 정보: source = "user_input"
- user_input 항목은 전체 업데이트 시에도 검색 대상 (링크를 찾아 보완하기 위해)

**last conference 처리 원칙**:
- last(past) edition은 AI 검색으로 찾지 않음 (API 절약)
- upcoming이 날짜가 지나면 자동으로 past로 전환됨 → 이것이 last를 채우는 주된 방법
- 그 외에는 수동 입력으로만 past edition 추가

### 3.3 기능 3: 신규 학회 발굴 (미래 구현)

> ⚠️ **이 기능은 1차 개발 범위에 포함되지 않음. 미래 확장으로 보류.**

**구상**: 사용자가 키워드(예: "열전달")를 입력하면, AI가 관련 키워드를 1차 제안하고, 해당 키워드군으로 기존 DB에 없는 학회를 웹에서 발굴. 가짜 학회(WASET 등) 필터링 포함.

### 4.2 페이지 1: 메인 테이블 뷰

| 컬럼 | 표시 | 정렬 | 필터 | 편집 |
|------|------|------|------|------|

| 기간 | 숫자 + "일" | ✅ | - | ✅ |
| 지역 | 텍스트 | ✅ | 드롭다운 | ✅ |

| 컬럼 | 표시 | 편집 |
|------|------|------|

| 종료일 | YYYY-MM-DD | ✅ |
| 장소 | 텍스트 | ✅ |
| 링크 | 링크 아이콘 | ✅ |

**참고 영역**:

| 컬럼 | 표시 | 편집 |

#### 4.2.3 액션 버튼

| 버튼 | 위치 | 동작 |
|------|------|------|
| 전체 업데이트 | 상단 툴바 | 페이지 2로 이동, 전체 업데이트 시작 |
| 전체 검증 | 상단 툴바 | 페이지 2로 이동, 전체 정합성 검증 시작 |
| 학회 추가 | 상단 툴바 | 빈 행 추가 또는 모달 폼 |
| 개별 업데이트 | 각 행 우측 | 해당 학회만 업데이트 → 페이지 2로 이동 |
| 개별 검증 | 각 행 우측 | 해당 학회만 검증 → 페이지 2로 이동 |
| 편집 | 각 행 우측 | 인라인 편집 모드 활성화 |
| 삭제 | 각 행 우측 | 확인 후 삭제 |
| API 키 설정 | 헤더 | 모달: Claude API 키 입력/변경 |
| 데이터 내보내기 | 헤더 | JSON 파일 다운로드 |
| 데이터 가져오기 | 헤더 | JSON 파일 업로드하여 DB 교체 |

#### 4.2.4 테이블 표시 규칙

- **기본 정렬**: upcoming 시작일 오름차순 (가까운 학회가 위)
- **upcoming 없는 학회**: 하단에 별도 그룹으로 표시
- **날짜가 지난 upcoming**: 자동으로 past로 전환 (빨간색 경고 표시 후 전환)
- **행 색상 구분**: 분류별 또는 분야별 미묘한 배경색 차이 (선택사항)

```
시스템: 당신은 학회/박람회 정보를 검색하는 전문 리서처입니다.

사용자: 다음 학회의 다음(upcoming) 개최 정보를 찾아주세요.
       
       학회명: {full_name}
       약칭: {abbreviation}
       주기: {cycle_years}년
       공식사이트: {official_url}
       마지막 개최: {last_edition_info}
       
       찾아야 할 정보:
       - 시작일 (YYYY-MM-DD)
       - 종료일 (YYYY-MM-DD)
       - 장소 (도시, 국가)
       - 공식 링크 (해당 회차 전용)
       
       JSON 형식으로 응답하세요.
```

### 5.4 정합성 검증용 프롬프트 구조 (개요)

```

사용자: 다음 학회의 기본 정보가 정확한지 검증해주세요.
       
       학회명: {full_name}
       약칭: {abbreviation}
       주기: {cycle_years}년
       기간: {duration_days}일
       지역: {region}
       공식사이트: {official_url}
       
       각 항목에 대해 "일치", "불일치 (올바른 값: ○○)", "확인 불가" 중 하나로 답하세요.
       JSON 형식으로 응답하세요.
```

### 5.5 API 비용 고려사항

- 학회 1건 업데이트 ≈ 웹 검색 포함 1회 API 호출
- 전체 업데이트 시 대상 학회 수만큼 호출 (순차적)
- 중단 버튼으로 불필요한 호출 방지
- pass 로직으로 이미 완성된 학회는 건너뛰어 비용 절감

### 6.2 접근 구조

```
열람자: URL 접속 → 학회 테이블 조회 (읽기 전용)

```

- 별도 로그인 없음
- API 키가 입력되어 있으면 관리자 기능 활성화
- API 키가 없으면 읽기 전용 모드

## 7. 미래 확장 로드맵

### 7.1 근미래 확장 (2단계)

| 기능 | 설명 |
|------|------|
| API 절약 필터 | 주기 기반 검색 조건 분기 (주기의 절반 미도래 시 일반 업데이트 제외) |

### 7.2 중장기 확장 (3단계)

| 기능 | 설명 |
|------|------|
| 신규 학회 자동 발굴 | 키워드 기반 AI 웹 검색으로 DB에 없는 학회 발굴 |

| 개인화 DB | 사용자별 마킹(중요도), 메모 분리 |
| 다중 사용자 편집 | 서버 도입 후 권한 관리 |
| 서버 기반 구조 전환 | API 키 서버 관리, 자동 GitHub 커밋 |
| 캘린더 뷰 | 학회 일정을 캘린더 형태로 시각화 |
| 분야 확장 | 열유체·건물공조 외 타 분야 DB 구축 |
